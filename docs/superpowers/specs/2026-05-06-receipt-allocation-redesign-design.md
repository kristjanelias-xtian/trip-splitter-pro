# Receipt Allocation Redesign — Design Spec

## Overview

Replace the current item-expansion-based allocation in `ReceiptReviewSheet` with a counter-based UI, add a By-Person alternative view, make carry-forward optional and OFF by default, and add inline translations for foreign-language receipts. Fixes a class of cascade bugs caused by mid-array expansion.

## Motivation

Three problems in the current flow:

1. **Cascade-after-expansion bug.** `expandItem()` splits a `qty=3` row into 3 inserted rows. The carry-forward cascade in `toggleParticipant`/`toggleAll` walks forward by index — after expansion, items that were originally below the multi-qty row sit at higher indices and get re-stomped. `mapped_items` is also persisted by array index, so re-edit mis-maps.
2. **Carry-forward is always on and surprises users.** Users selecting people for one item didn't expect downstream items to change.
3. **Item-first only.** Doesn't fit large restaurant bills (12 people, ~5 items) where pivoting to person-first is the natural mental model.

Plus a feature gap: foreign-language menus are unreadable for users who don't speak the local language.

## Design Decisions (settled in brainstorm)

| Decision | Choice |
|---|---|
| Multi-qty UI | **Counter/stepper, no expansion.** |
| View mode | **Segmented `By Item | By Person` toggle**, same underlying state. |
| Carry-forward default | **OFF**, optional toggle in By-Item view, persisted per user. |
| Translation | **Always-on**, LLM returns `name` + `nameOriginal`, UI shows both inline when they differ. |
| Target language | **Browser locale** (`navigator.language`), no UI selector. |

## Data Shape

Single source of truth, view-agnostic:

```ts
type ItemId = string

type ReceiptItem = {
  id: ItemId                   // stable, generated by edge function
  name: string                 // localized to target language
  nameOriginal: string         // verbatim from receipt
  price: number                // total for the line, not per-unit
  qty: number
  manuallySet: boolean         // for carry-forward gating
}

type Allocations = Map<ItemId, Map<ParticipantId, number>>
//                  ^itemId   ^who got how many units of that item
```

**Invariant:** `sum(counts) <= item.qty` for every item. Going over is blocked at the UI; under is allowed mid-edit but blocks submit.

**Distribution math:** `pricePerUnit = item.price / item.qty`. Each participant's share for a line = `count * pricePerUnit`. Sum over all items → final per-person totals fed into `buildDistribution`.

## UI — By-Item View

Each item is one row, regardless of qty.

```
+---------------------------------------------+
|  Cabbage soup                          24   |
|  Borscht                          (italic)  |  <- only if name != nameOriginal
|                                             |
|  3 of 8 assigned                            |  <- progress chip
|                                             |
|  [Joe -1+]  [John -2+]  [Mary -0+]   ...    |
|  [+ Everyone equally]                       |
+---------------------------------------------+
```

- **Stepper** per participant: `-` decrement disabled at 0, `+` disabled when item is full (`assigned == qty`).
- For `qty == 1`, the stepper collapses to a tap-to-toggle chip — same density as the current UI on simple receipts.
- **"Everyone equally"** distributes qty across all current participants: floor across, remainder lumped on first participant alphabetically (deterministic).
- **Progress chip:** amber while `0 < assigned < qty`, green at `assigned == qty`, hidden when `assigned == 0`.
- **Inline name editing** preserved.
- **No expansion.** `expandItem()` deleted.

## UI — By-Person View

Same data, pivoted. Each participant is a row:

```
+---------------------------------------------+
|  Joe                        18.50    >    |  <- collapsed
|  3 items                                    |
+---------------------------------------------+
|  John                       11.00    >    |
|  2 items                                    |
+---------------------------------------------+
```

Tap to expand a person:

```
  v Joe                                 18.50
    Cabbage soup        2 left   [-  1  +]
    Beer                1 left   [-  2  +]
    Bread              fully assigned  (-)
```

- **"X left"** = `item.qty - sum(allAllocationsForItem)`. Disables `+` at 0.
- **Sheet header chip** shows global progress: "10 of 12 units assigned".
- **No carry-forward in this view** — concept doesn't apply. Toggle is hidden.
- Translated names rendered the same way as By-Item view.

**View persistence:** localStorage key `spl1t:receipt-allocation-view`, default `'by-item'`. Switching never mutates allocation state — both views render the same `Allocations` map.

## Carry-Forward Toggle (By-Item only)

Header layout:

```
By Item | By Person                [Carry forward: OFF]
```

- **Default OFF.** localStorage key `spl1t:receipt-carry-forward`, persisted per user.
- **When ON:** every change to item N's allocations triggers `applyCarryForward(itemId)`. For every following item M (in display order) where `M.manuallySet === false`: prefill participant set from N's set, distribute M's qty equally (floor + remainder lumped on first participant alphabetically).
- **Direct user interaction with an item (clicking +/-, the chip, "Everyone equally", inline edit) flips its `manuallySet` to true** — once a user has manually edited an item, it's frozen against future cascade. Receiving an allocation *via* carry-forward does NOT flip the flag — those items remain eligible to be re-prefilled if an earlier item changes again.
- **When OFF, no cascade ever runs.** Flipping the toggle OFF mid-session does NOT clear previously-cascaded values; it just stops further auto-fill.
- **Hidden in By-Person view.**

## Translation

### Edge function changes (`process-receipt`, `process-kopikas-receipt`)

Request body gains:
```ts
targetLanguage: string  // e.g. 'en', 'et', 'fr'
```

Response schema for each item:
```ts
{
  id: string             // crypto.randomUUID() generated by the edge function
  name: string           // translated/normalized into targetLanguage
  nameOriginal: string   // verbatim from receipt (Latinized only if non-Latin script)
  price: number
  qty: number
}
```

System prompt additions (~5 lines):
- Receive `targetLanguage` from caller.
- For each item, return `nameOriginal` exactly as printed (preserving existing romanization for non-Latin scripts).
- Return `name` translated/normalized into `targetLanguage`. If already in target language, both fields are identical.
- The model decides per-item whether translation is meaningful — no separate detection pass.

### Caller change

`QuickScanCreateFlow` (and any other invoker of `process-receipt`) passes:
```ts
targetLanguage: navigator.language?.split('-')[0] ?? 'en'
```

### UI display rule

- If `nameOriginal !== name`: primary line = `name` (top, bigger), secondary line = `nameOriginal` (muted italic underneath).
- If equal: render one line.

### Storage and backward compat

- `receipt_tasks.extracted_items` is JSONB and open-schema. New fields ride along. No migration.
- Stored `nameOriginal`/`name` are reused on re-edit. No re-translation.
- Old rows with only `name` are read with `nameOriginal` defaulting to `name` (no UI change).

## `mapped_items` Persistence (Re-edit Round-Trip)

### New shape (`src/types/receipt.ts`)

```ts
type MappedItem = {
  itemId: string
  participantCounts: Array<{ participantId: string; count: number }>
}
```

- Keyed by stable `itemId`, never array index. Reordering or deleting items can't desync mappings.
- Per-person counts replace participant sets.

### Load adapter (in `ReceiptContext`)

When loading a `mapped_items` row, detect old shape (`item_index` + `participant_ids`) and convert:
- `item_index` -> look up `extracted_items[item_index].id` (or generate one in-memory if old extracted_items lack ids).
- `participant_ids: string[]` -> `participantCounts: [{participantId, count: 1}, ...]` (old model had qty=1 post-expansion).

One-time per-load adapter, no DB migration script. New writes use the new shape.

### `manuallySet` on re-edit

Recomputed from data: any item with `count > 0` for any participant is `manuallySet: true`.

## Bug Fixes Summary

- **Expansion-cascade bug:** gone by construction — no more expansion.
- **Index-based persistence:** gone — `itemId` keys.
- **Submit-time validation:** new. Block submit when any item has `assigned < qty`. Error message: "X of Y units unassigned on Cabbage soup". The current expanded-rows model silently accepts under-assignment.
- **Cascade code (lines 268–304 of `ReceiptReviewSheet.tsx`)** deleted. Replaced with a single `applyCarryForward(itemId)` helper that runs only when the toggle is ON.

## Out of Scope

- Real-time multi-user receipt allocation (single-user per session).
- Per-unit price overrides (one beer at a different price). Workaround: split into two items manually.
- Translation language selector UI — locale-derived only.
- Changing the LLM model.
- Changing the persisted DB schema for `receipt_tasks` — all changes ride inside JSONB columns.

## Risks

- **LLM cost increase from always returning `nameOriginal` + `name`.** Marginal — receipts are short and the additional field is a few tokens per item.
- **localStorage key collisions** with existing keys. Both new keys are namespaced `spl1t:receipt-*` — no collisions in current code.
- **Backward-compat adapter complexity** for old `mapped_items`. Mitigated by keeping adapter small and tested. Risk that production has rows without `id` on `extracted_items` — adapter generates synthetic ids in-memory for legacy rows; new writes always have stable ids.
- **Submit-time validation is stricter than today.** Users currently can save under-allocated receipts (the expanded-rows model lets you skip an expanded sub-item). The validation could surprise existing power users — mitigated by clear error message and progress chip preview before submit.

## Acceptance Criteria

- A 3-beer item never desyncs other items' allocations under any sequence of edits.
- Toggling between By Item and By Person preserves all allocations losslessly.
- With carry-forward OFF (default), allocating item 1 has zero effect on item 2.
- With carry-forward ON, allocating item 1 prefills item 2 (and onward) with the same participant set + equal qty distribution, and prior manual edits to those items are preserved.
- A receipt scanned in French shows French original + English translation (for an `en` browser) inline.
- Re-editing a receipt saved before this change loads with prior allocations intact.
- Submit blocks with a clear error when any item is under-assigned.
